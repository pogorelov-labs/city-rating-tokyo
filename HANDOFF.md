# CRTKY-109 — Description Generation Handoff

**Goal:** Generate multilingual (EN/JA/RU) neighborhood descriptions for **all 1493 Tokyo stations**.

**Scope note (2026-04-15):** previously we planned to preserve ~252 single-field RU descriptions from AI-researched stations. That decision was reversed — those stations are regenerated too, giving the whole dataset a single consistent voice and the same 4-field structure `{atmosphere, landmarks, food, nightlife}` × `{en, ja, ru}`. Legacy RU text stays in git history until the merge step overwrites `demo-ratings.ts`.

**Tools that can run this in parallel:** Claude Code, Codex (OpenAI CLI), Cursor, any LLM agent that can read files and write files. Each tool picks work off the queue without conflicts.

---

## Quick start (for any LLM agent)

```
You are part of a parallel generation pool for Tokyo station descriptions.

1. Read data/prompts/<slug>.md — it contains everything you need (voice, rules, few-shot examples, station data).
2. Produce the JSON output following the contract at the bottom of that file.
3. Save it to data/descriptions/<slug>.json (pretty-printed JSON, UTF-8).
4. DO NOT edit any other file.
5. DO NOT generate for slugs that already have a file in data/descriptions/.

To find your next task:
    python3 scripts/queue-status.py --next 10

Pick any slug from that list that doesn't already exist as data/descriptions/<slug>.json.
If multiple agents race, last writer wins — that's fine.
```

## File layout

```
data/
  station-datamart.json           # canonical per-station data (regenerable, gitignored)
                                  # generated by: python3 scripts/build-datamart.py
  prompts/
    <slug>.md                     # INPUT — self-contained prompt per station (1235 files)
  descriptions/
    <slug>.json                   # OUTPUT — generated description for one station
                                  # shape: {en: {...}, ja: {...}, ru: {...}}
  generated-descriptions.json     # merged output (run merge script at the end)

research/
  description-generation-rules.md # the rules doc (duplicated in each prompt)

scripts/
  build-datamart.py               # rebuild data/station-datamart.json from NocoDB + local JSON
  build-prompts-dir.py            # regenerate data/prompts/ from datamart (idempotent)
  queue-status.py                 # show progress + next to generate
  merge-descriptions.py           # merge data/descriptions/*.json → generated-descriptions.json
```

## Output contract

Each `data/descriptions/<slug>.json` must contain:

```json
{
  "en": {
    "atmosphere": "string, 40-80 chars ideal",
    "landmarks":  "string, 40-80 chars ideal",
    "food":       "string, 40-80 chars ideal",
    "nightlife":  "string, 30-60 chars ideal"
  },
  "ja": { "atmosphere": "...", "landmarks": "...", "food": "...", "nightlife": "..." },
  "ru": { "atmosphere": "...", "landmarks": "...", "food": "...", "nightlife": "..." }
}
```

**No wrapping, no metadata, no markdown fences in the file.** Just the raw JSON object.

## Concurrency model

- **Work claim = file existence.** An agent picks a slug where `data/descriptions/<slug>.json` does **not** exist.
- **Race OK.** If two agents produce output for the same slug simultaneously, the last writer wins. No lock file needed.
- **Resumable.** Kill and restart any agent any time — it just picks up the next unclaimed slug.
- **Idempotent.** Re-running the pipeline is safe; it only processes missing files.

## Voice & quality bar

See `data/prompts/<slug>.md` — every prompt includes 3 few-shot examples (data-rich / data-sparse / mid-tier) to lock in the voice. You MUST match:

- **Telegram-style** — comma-separated phrases, not full tourist-brochure sentences.
- **Specific, not generic.** "Cheap izakaya, charcoal yakitori, sushi counters" beats "variety of restaurants".
- **Honest when data is thin.** "Data limited. Quiet office street." is better than making things up.
- **Trade-offs welcome.** "Great food but gritty after dark."

Rejected outputs (auto-detected by `queue-status.py --failed`):
- Missing language or field
- Empty strings
- Not valid JSON
- Obvious LLM boilerplate ("This is a great neighborhood where...")

## Run modes

### Claude Code (in this repo)

Use the Agent tool with `model: haiku` and paste the contents of `data/prompts/<slug>.md` as the prompt. Save response to `data/descriptions/<slug>.json`. ~7 seconds per station with haiku; can run 5+ in parallel.

### Codex CLI (OpenAI) — local

```bash
# Example wrapper
cat data/prompts/<slug>.md | codex chat --model gpt-5-mini > data/descriptions/<slug>.json
```

Strip any markdown fences the model adds with `jq . data/descriptions/<slug>.json` before committing.

### Cursor — local

Open `data/prompts/<slug>.md` in Cursor, ask "produce the JSON for this prompt and save to `data/descriptions/<slug>.json`", accept.

### Loop script (any CLI LLM)

```bash
for prompt in data/prompts/*.md; do
  slug=$(basename "$prompt" .md)
  out="data/descriptions/$slug.json"
  if [ -f "$out" ]; then continue; fi       # skip done
  your-llm-cli --prompt-file "$prompt" | jq . > "$out"
  if [ ! -s "$out" ] || ! jq empty "$out" 2>/dev/null; then
    echo "FAIL: $slug" >&2
    rm -f "$out"                             # leave it missing so someone retries
  fi
done
```

## Data signals included in each prompt

Every `data/prompts/<slug>.md` bundles:

- **Identity:** name, kanji, prefecture, city/ward
- **Transport:** line names + operators, line_count, daily_passengers, avg_transit_min, last_train time
- **Ratings (1-10):** all 9 categories with confidence (strong/moderate/estimate/editorial)
- **Food:** HotPepper counts (izakaya/bars/cafes/total), OSM food_count
- **Nightlife:** midnight venues, karaoke, clubs, hostels
- **Green:** park count + total area in sqm
- **Vibe:** cultural venues, pedestrian streets
- **Crime:** total, per-10k rate (Tokyo only, 615 stations)
- **Environment:** elevation_m + tier, seismic risk tier
- **Rent:** 1K-1LDK in yen + source (suumo/estat)
- **Essentials:** supermarket/pharmacy/clinic counts

If a signal is missing, the field is omitted from the prompt (don't invent it).

## Troubleshooting

**"Prompts don't exist yet"** — run `python3 scripts/build-prompts-dir.py` (generates from current datamart).

**"Datamart is stale"** — run `python3 scripts/build-datamart.py` (~60s, hits NocoDB live).

**"Some outputs look wrong"** — run `python3 scripts/queue-status.py --failed`, delete the bad files, re-queue.

**"I want to regenerate a specific slug"** — delete `data/descriptions/<slug>.json` and run the generation again.

**"All my outputs have markdown fences"** — your LLM wraps JSON in ```` ```json ... ``` ````; strip them. `jq . file.json` will fail on wrapped output and succeed on raw.

## Files that are OK to commit

- `data/descriptions/*.json` — YES, commit each
- `data/generated-descriptions.json` — YES, commit the merged file
- `data/prompts/*.md` — OPTIONAL (regenerable; can add to .gitignore)
- `data/station-datamart.json` — NO (gitignored, regenerable, 5 MB)

## Files that are NOT OK to edit

- `app/src/data/demo-ratings.ts` — don't touch during generation; that's a separate PR step.
- Anything in `app/src/` — stay away.
- `data/stations.json` — canonical slug source; read-only.
